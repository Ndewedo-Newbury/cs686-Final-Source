## App Architecture

### Services

Three independent microservices, each owning its own database:

| Service | Port (local) | Database | Responsibility |
|---|---|---|---|
| Auth Service | 8001 | `auth_db` | Register, login, JWT issuance |
| Workouts Service | 8002 | `workouts_db` | Log and manage workouts/exercises |
| Analytics Service | 8003 | `analytics_db` | Compute stats, consume SQS events |

### Tech Stack

- **Language/Framework:** Python 3.11 + FastAPI
- **Database:** Amazon RDS (PostgreSQL) — single RDS instance, separate database per service
- **Schema Migrations:** Alembic (expand/contract pattern for zero-downtime migrations)
- **Message Queue:** Amazon SQS — async event passing between Workouts → Analytics
- **Frontend:** React (TypeScript) SPA hosted on S3 + CloudFront

### Service Communication

**REST + SQS Hybrid:**
- Synchronous REST for time-sensitive operations (auth, fetching workouts)
- Async SQS for event-driven operations (workout logged → analytics update)

SQS events carry typed full payloads (no follow-up REST calls needed):
```json
{
  "event": "workout.logged",
  "schema_version": "1.0",
  "user_id": "...",
  "workout_id": "...",
  "exercise_count": 3,
  "logged_at": "2026-04-20T10:00:00Z"
}
```

### Authentication

- Auth Service issues signed JWTs on login/register
- Workouts and Analytics Services validate JWTs **locally** using a shared secret via the `shared/auth/jwt.py` library
- No network hop per request — fast, no Auth Service bottleneck
- Token expiry: 60 minutes

### API Design

- Versioned REST endpoints: `/api/v1/...`
- OpenAPI/Swagger auto-generated by FastAPI at `/docs` per service
- No API versioning middleware needed at this stage (single frontend client)

### Database Schema

**auth_db**
```
users: id, email, password_hash, created_at
```

**workouts_db**
```
workouts:  id, user_id, name, logged_at
exercises: id, workout_id, name, sets, reps, weight_kg
```

**analytics_db**
```
user_stats: id, user_id, total_workouts, current_streak, last_workout_at, updated_at
```

> Cross-service references (user_id, workout_id) are plain UUIDs — no cross-database foreign keys. Integrity enforced at the application/API level.

### Monorepo Structure

```
fitness-tracker/
├── shared/                    ← JWT, DB base, SQS event schemas
├── backend/
│   ├── auth-service/
│   ├── workouts-service/
│   └── analytics-service/
├── frontend/                  ← React TypeScript SPA
├── terraform/
│   ├── bootstrap/             ← S3 state bucket + DynamoDB lock (run once)
│   ├── modules/               ← reusable modules (eks, rds, sqs, vpc, etc.)
│   └── environments/          ← dev / qa / uat / prod
├── helm/                      ← Helm charts + per-environment values files
├── k8s/
│   ├── argocd/
│   ├── monitoring/
│   └── network-policies/
└── scripts/
    ├── init-db.sql
    ├── localstack/init.sh
    └── bootstrap.sh
```

### AWS Infrastructure

| Component | Service | Notes |
|---|---|---|
| Container orchestration | EKS (Kubernetes) | Managed node groups |
| API entry point | AWS API Gateway | Single URL, path-based routing to all services |
| Frontend hosting | Nginx on EKS | Multi-stage Docker build, served as a pod alongside backend services |
| SSL certificates | AWS ACM | Auto-renewing; one cert on API Gateway |
| Container registry | ECR | One repository per service + frontend |
| Secrets | AWS Secrets Manager + External Secrets Operator | ESO syncs secrets into Kubernetes |
| Admin access | SSM Session Manager | No Bastion host, no open SSH ports |

### Network Security

- **VPC** with public/private subnet separation
  - Public: ALB, API Gateway endpoints
  - Private: EKS nodes, RDS, SQS VPC endpoints
- **Kubernetes NetworkPolicies** enforce pod-to-pod microsegmentation — a compromised service cannot reach another service's database
- **IRSA (IAM Roles for Service Accounts)** — each service has its own IAM role with least-privilege permissions (its own DB + secrets only)

## Deployment & CI/CD

### Environments

| Environment | Purpose | RDS size | EKS nodes |
|---|---|---|---|
| dev | Active development | db.t3.micro | 1 |
| qa | Nightly automated testing | db.t3.micro | 1 |
| uat | Stakeholder validation | db.t3.small | 2 |
| prod | Live | db.t3.medium | 3+ |

### Promotion Flow

```
Dev push
  → GitHub Actions: lint → unit tests → integration tests → build image → push to ECR → deploy to dev (rolling)

2AM nightly (cron)
  → Promote dev image → QA → run E2E tests (pytest + Playwright)

Conventional commit (RC1, RC2, ...) or PR merge to main
  → Promote QA-approved image → UAT → smoke tests

git tag v1.0.1 + git push origin v1.0.1
  → Run Alembic migrations on prod RDS
  → ArgoCD Blue/Green: deploy Green (new version)
  → Health checks pass → swap 100% traffic Blue → Green
  → Blue pods drain (15 min) → terminate
  → GitHub Release created
```

> Manual "click-to-deploy" in the AWS Console is prohibited. All promotions are git-event driven.

### CI/CD Tooling

- **GitHub Actions** — CI pipeline (test, build, push ECR, image tagging)
- **ArgoCD** — GitOps CD; watches the repo and syncs Helm chart changes to EKS
- **Helm** — Kubernetes manifest templating; one chart per service, one values file per environment

### Deployment Strategy

- **Dev / QA / UAT:** Kubernetes rolling updates (`maxUnavailable: 0`, readiness probe gated)
- **Production:** Blue/Green deployment via ArgoCD Rollouts
  - Green is deployed and health-checked before traffic switches
  - Blue kept alive for 15 minutes as instant rollback target
  - All Alembic migrations run before any pod is replaced

### Image Tagging Strategy (ECR)

Each image carries multiple tags to track its promotion state:

```
auth-service:a1b2c3d       ← immutable commit SHA
auth-service:dev-latest    ← latest dev build
auth-service:qa-latest     ← latest QA-promoted build
auth-service:RC1           ← release candidate
auth-service:v1.0.1        ← production release
```

### Terraform State

- **Backend:** S3 bucket + DynamoDB table for state locking
- **Structure:** Separate state file per environment (dev/qa/uat/prod)
- **Bootstrap:** One-time manual run of `terraform/bootstrap/` creates the S3 bucket and DynamoDB table before all other Terraform runs

### Testing Strategy

| Layer | Tool | When it runs | Environment |
|---|---|---|---|
| Unit | pytest | Every dev push | GitHub Actions runner |
| Integration | pytest + Docker PostgreSQL | Every dev push | GitHub Actions runner |
| E2E (backend) | pytest | After nightly QA promotion | Live QA environment |
| E2E (frontend) | Playwright | After nightly QA promotion | Live QA environment |

> E2E tests run against the live QA environment — real services, real RDS, real SQS. Test data is managed via pytest fixtures (setup/teardown per test).

### Zero Downtime Guarantees

| Concern | Solution |
|---|---|
| Rolling AMI updates | ALB connection draining + ASG rolling replacement |
| Pod replacement | `maxUnavailable: 0` + readiness probes |
| Production deploys | Blue/Green — no traffic switch until Green is healthy |
| Schema changes | Alembic expand/contract — additive migrations before deploy, destructive after |
| SQS during deploy | Messages requeue after visibility timeout if consumer restarts |

## Observability & Logging

### Metrics & Dashboards

- **Prometheus** — scrapes metrics from all EKS nodes and services
- **Grafana** — dashboards for CPU, memory, and disk space per node
- **Deployment:** `kube-prometheus-stack` Helm chart on EKS (includes Prometheus, Grafana, Alertmanager, Node Exporter, kube-state-metrics)

### Alerting

- **Alertmanager** (included in `kube-prometheus-stack`) routes alerts to:
  - Email via SMTP
  - Slack via webhook
- Alerts fire on critical resource thresholds (CPU, memory, disk)

### Grafana Access & Security

- Grafana is externally accessible via AWS Load Balancer Controller + ACM certificate (HTTPS)
- **Username/password login is disabled**
- Authentication via **GitHub OAuth** — users must authenticate with a GitHub account
- Configured in Grafana Helm values (`grafana.ini` auth.github block)

### Centralized Logging

- **Loki** — log aggregation (label-indexed, not full-text)
- **Promtail** — DaemonSet on every EKS node, ships logs from all 3 service pods to Loki
- **Grafana** — unified UI for both metrics (Prometheus) and logs (Loki)

All 3 services log to stdout. Kubernetes captures stdout. Promtail collects by label.

Cross-service log query example (LogQL):
```
{app=~"auth-service|workouts-service|analytics-service"} |= "ERROR"
```

Per-service query:
```
{app="workouts-service"} | json | level="ERROR"
```

## Local Development

### Prerequisites

- Docker + Docker Compose

### Run

```bash
docker compose up --build
```

This starts:
- PostgreSQL (3 databases: auth_db, workouts_db, analytics_db)
- LocalStack (mocks AWS SQS locally)
- All 3 FastAPI services with hot reload

### Service URLs

| Service | URL | Swagger |
|---|---|---|
| Auth | http://localhost:8001 | http://localhost:8001/docs |
| Workouts | http://localhost:8002 | http://localhost:8002/docs |
| Analytics | http://localhost:8003 | http://localhost:8003/docs |

### End-to-End Test Flow

```
1. POST /api/v1/auth/register        → { access_token }
2. POST /api/v1/workouts/            → logs workout, publishes to SQS (Bearer <token>)
3. (Analytics service consumes SQS event in background)
4. GET  /api/v1/analytics/stats      → { total_workouts, current_streak } (Bearer <token>)
```
