## App Architecture

### Services

Three independent microservices, each owning its own database:

| Service | Port (local) | Database | Responsibility |
|---|---|---|---|
| Auth Service | 8001 | `auth_db` | Register, login, JWT issuance |
| Workouts Service | 8002 | `workouts_db` | Log and manage workouts/exercises |
| Analytics Service | 8003 | `analytics_db` | Compute stats, consume SQS events |

### Tech Stack

- **Language/Framework:** Python 3.11 + FastAPI
- **Database:** Amazon RDS (PostgreSQL) — single RDS instance, separate database per service
- **Schema Migrations:** Alembic — each service runs `alembic upgrade head` on startup; the expand/contract pattern keeps migrations backward-compatible so old and new pods can run simultaneously during a rolling update
- **Message Queue:** Amazon SQS — async event passing between Workouts → Analytics
- **Frontend:** React (TypeScript) SPA served by Nginx, running as a pod in EKS

### Service Communication

**REST + SQS Hybrid:**
- Synchronous REST for time-sensitive operations (auth, fetching workouts)
- Async SQS for event-driven operations (workout logged → analytics update)

SQS events carry typed full payloads (no follow-up REST calls needed):
```json
{
  "event": "workout.logged",
  "schema_version": "1.0",
  "user_id": "...",
  "workout_id": "...",
  "exercise_count": 3,
  "logged_at": "2026-04-20T10:00:00Z"
}
```

### Authentication

- Auth Service issues signed JWTs on login/register
- Workouts and Analytics Services validate JWTs **locally** using a shared secret via the `shared/auth/jwt.py` library
- No network hop per request — fast, no Auth Service bottleneck
- Token expiry: 60 minutes

### API Design

- Versioned REST endpoints: `/api/v1/...`
- OpenAPI/Swagger auto-generated by FastAPI at `/docs` per service
- Single entry point: AWS API Gateway with path-based routing to all backend services

### Database Schema

**auth_db**
```
users: id, email, password_hash, created_at
```

**workouts_db**
```
workouts:  id, user_id, name, logged_at
exercises: id, workout_id, name, sets, reps, weight_kg
```

**analytics_db**
```
user_stats: id, user_id, total_workouts, current_streak, last_workout_at, updated_at
```

> Cross-service references (user_id, workout_id) are plain UUIDs — no cross-database foreign keys. Integrity enforced at the application/API level.

### AWS Infrastructure

| Component | Service | Notes |
|---|---|---|
| Container orchestration | EKS (Kubernetes) | Managed node group, t3.small instances (3 desired / 2 min / 4 max) |
| API entry point | AWS API Gateway | Single URL, path-based routing to all services |
| Frontend hosting | Nginx on EKS | Multi-stage Docker build; served as a pod alongside backend services |
| SSL certificates | AWS ACM | Auto-renewing; one cert on API Gateway |
| Container registry | ECR | One repository per service + frontend |
| Secrets | AWS Secrets Manager + External Secrets Operator | ESO syncs secrets into Kubernetes |
| Admin access | SSM Session Manager | No Bastion host, no open SSH ports |

### Network Security

- **VPC** with public/private subnet separation
  - Public: ALB, API Gateway endpoints
  - Private: EKS nodes, RDS
- **Kubernetes NetworkPolicies** enforce pod-to-pod microsegmentation — a compromised service cannot reach another service's database
- **IRSA (IAM Roles for Service Accounts)** — each service pod uses the cluster's LabRole with least-privilege permissions scoped via Secrets Manager paths

## Deployment & CI/CD

### Environments

Only the **dev** environment is deployed. The Helm charts and ApplicationSet are structured to support multiple environments (qa, uat, prod) — values files and Terraform modules for those environments exist in the repo — but only dev is actively provisioned on AWS.

| Environment | Status | RDS instance | EKS nodes |
|---|---|---|---|
| dev | **Active** | db.t3.micro | t3.small × 3 |
| qa / uat / prod | Not deployed | — | — |

### CI/CD Flow

```
Push to main branch
  → GitHub Actions:
      lint
      unit tests (pytest)
      integration tests (pytest + Docker PostgreSQL)
      build Docker images
      push to ECR (tagged sha-<commit>)
      update dev-*.yaml image tags
      commit + push tag update back to repo
  → ArgoCD detects new tag in values file
  → ArgoCD syncs Helm chart to EKS dev namespace
  → Rolling update: new pods start, pass readiness probe, old pods terminate
```

Manual "click-to-deploy" in the AWS Console is prohibited. All deployments are git-event driven.

### CI/CD Tooling

- **GitHub Actions** — CI pipeline (test, build, push ECR, image tagging)
- **ArgoCD** — GitOps CD; watches the repo and syncs Helm chart changes to EKS
- **Helm** — Kubernetes manifest templating; one chart per service, one values file per environment

### Deployment Strategy

**Dev (current):** Kubernetes rolling update
- `maxUnavailable: 1` — one pod replaced at a time
- Readiness probe (`/health`) gates each pod before the next is replaced
- Alembic migrations run on pod startup before the service begins accepting requests

**Blue/Green (built-in, not active in dev):** Each Helm chart conditionally renders an ArgoCD `Rollout` resource instead of a standard `Deployment` when `blueGreen.enabled: true` is set in the values file. With blue/green enabled:
- ArgoCD Rollouts deploys the new (green) version alongside the current (blue) version
- Green receives no live traffic until health checks pass
- Traffic switches to green only after promotion (manual or automated)
- Blue pods drain for 15 minutes before termination
- This would be activated in a prod environment by setting `blueGreen.enabled: true` in `values/prod-<service>.yaml`

### Image Tagging Strategy (ECR)

CI tags each image with the commit SHA:
```
auth-service:sha-a1b2c3d    ← immutable commit SHA (what's deployed)
auth-service:latest         ← updated on every successful CI run
```

The dev values files (`DevOps/helm/values/dev-*.yaml`) are updated by CI to point to the new SHA tag. ArgoCD detects this change and triggers a sync.

### Terraform State

- **Backend:** S3 bucket (`fitness-tracker-terraform-state-010525338060`) with S3-native state locking
- **Structure:** Separate state file per environment (`dev/terraform.tfstate`)
- **Bootstrap:** One-time manual run of `terraform/bootstrap/` creates the S3 bucket before all other Terraform runs

### Testing Strategy

| Layer | Tool | When it runs | Environment |
|---|---|---|---|
| Unit | pytest | Every push | GitHub Actions runner |
| Integration | pytest + Docker PostgreSQL | Every push | GitHub Actions runner |
| Lambda integration | pytest (via test-runner Lambda) | Post-deploy | Live dev RDS + SQS |

## Observability & Logging

### How Monitoring Works

```
EKS Nodes
  └── node-exporter DaemonSet        ← exposes CPU/memory/disk metrics on :9100
  └── kube-state-metrics Deployment  ← exposes Kubernetes object metrics

Prometheus (StatefulSet)
  └── scrapes node-exporter + kube-state-metrics every 15s via ServiceMonitors
  └── evaluates PrometheusRules → fires alerts to Alertmanager
  └── retains 15 days of metrics on a 20 Gi EBS volume

Alertmanager
  └── routes alerts → AWS SNS topic → email subscribers

Grafana (Deployment, https://grafana.cs686.live)
  └── Prometheus datasource → metrics dashboards
  └── Loki datasource        → log explorer

Loki (StatefulSet)
  └── receives logs from Promtail DaemonSet (one pod per node)
  └── Promtail ships stdout of all pods via Kubernetes log discovery
```

All components are deployed via the `kube-prometheus-stack` Helm chart plus a standalone Loki Helm chart, both in the `monitoring` namespace.

### Metrics & Dashboards

- **Prometheus** — scrapes metrics from all EKS nodes and services via `ServiceMonitor` CRDs
- **Grafana** — pre-provisioned dashboard: **Infrastructure → node-exporter-full** (gnetId 1860) shows per-node CPU, memory, disk, and network
- **Grafana access:** https://grafana.cs686.live — login via GitHub OAuth (username/password login disabled)

### Alert Thresholds

Defined in `DevOps/k8s/monitoring/node-alerts.yaml` as a `PrometheusRule`. All alerts must persist for 5 minutes before firing.

| Metric | Warning | Critical |
|---|---|---|
| CPU usage | > 80% | > 90% |
| Memory usage | > 85% | > 95% |
| Disk usage (/) | > 80% | > 90% |

- **Warning** alerts go to the `default` Alertmanager receiver (SNS → email)
- **Critical** alerts go to the `critical` Alertmanager receiver (same SNS topic, separate routing)

To subscribe an email address to alerts:
```bash
aws sns subscribe \
  --topic-arn arn:aws:sns:us-west-2:010525338060:fitness-tracker-dev-alerts \
  --protocol email \
  --notification-endpoint <your-email> \
  --region us-west-2
```
AWS sends a confirmation email — click the link to activate.

To apply the alert rules to the cluster:
```bash
kubectl apply -f DevOps/k8s/monitoring/node-alerts.yaml
```

### Centralized Logging

- **Loki** — log aggregation (label-indexed, not full-text), stores logs on a 10 Gi EBS volume
- **Promtail** — DaemonSet on every EKS node, ships stdout of all pods to Loki
- **Grafana Explore** — query logs alongside metrics in the same UI

All 3 services log to stdout. Kubernetes captures stdout. Promtail collects by pod label.

Cross-service log query (LogQL):
```
{app=~"auth-service|workouts-service|analytics-service"} |= "ERROR"
```

Per-service query:
```
{app="workouts-service"} | json | level="ERROR"
```
