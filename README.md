## App Architecture

### Services

Three independent microservices, each owning its own database:

| Service | Port (local) | Database | Responsibility |
|---|---|---|---|
| Auth Service | 8001 | `auth_db` | Register, login, JWT issuance |
| Workouts Service | 8002 | `workouts_db` | Log and manage workouts/exercises |
| Analytics Service | 8003 | `analytics_db` | Compute stats, consume SQS events |

### Tech Stack

- **Language/Framework:** Python 3.11 + FastAPI
- **Database:** Amazon RDS (PostgreSQL) — single RDS instance, separate database per service
- **Schema Migrations:** Alembic (expand/contract pattern for zero-downtime migrations)
- **Message Queue:** Amazon SQS — async event passing between Workouts → Analytics
- **Frontend:** React (TypeScript) SPA hosted on S3 + CloudFront

### Service Communication

**REST + SQS Hybrid:**
- Synchronous REST for time-sensitive operations (auth, fetching workouts)
- Async SQS for event-driven operations (workout logged → analytics update)

SQS events carry typed full payloads (no follow-up REST calls needed):
```json
{
  "event": "workout.logged",
  "schema_version": "1.0",
  "user_id": "...",
  "workout_id": "...",
  "exercise_count": 3,
  "logged_at": "2026-04-20T10:00:00Z"
}
```

### Authentication

- Auth Service issues signed JWTs on login/register
- Workouts and Analytics Services validate JWTs **locally** using a shared secret via the `shared/auth/jwt.py` library
- No network hop per request — fast, no Auth Service bottleneck
- Token expiry: 60 minutes

### API Design

- Versioned REST endpoints: `/api/v1/...`
- OpenAPI/Swagger auto-generated by FastAPI at `/docs` per service
- No API versioning middleware needed at this stage (single frontend client)

### Database Schema

**auth_db**
```
users: id, email, password_hash, created_at
```

**workouts_db**
```
workouts:  id, user_id, name, logged_at
exercises: id, workout_id, name, sets, reps, weight_kg
```

**analytics_db**
```
user_stats: id, user_id, total_workouts, current_streak, last_workout_at, updated_at
```

> Cross-service references (user_id, workout_id) are plain UUIDs — no cross-database foreign keys. Integrity enforced at the application/API level.

### Monorepo Structure

```
fitness-tracker/
├── shared/                    ← JWT, DB base, SQS event schemas
├── backend/
│   ├── auth-service/
│   ├── workouts-service/
│   └── analytics-service/
├── frontend/                  ← React TypeScript SPA
├── terraform/
│   ├── bootstrap/             ← S3 state bucket + DynamoDB lock (run once)
│   ├── modules/               ← reusable modules (eks, rds, sqs, vpc, etc.)
│   └── environments/          ← dev / qa / uat / prod
├── helm/                      ← Helm charts + per-environment values files
├── k8s/
│   ├── argocd/
│   ├── monitoring/
│   └── network-policies/
└── scripts/
    ├── init-db.sql
    ├── localstack/init.sh
    └── bootstrap.sh
```

### AWS Infrastructure

| Component | Service | Notes |
|---|---|---|
| Container orchestration | EKS (Kubernetes) | Managed node groups |
| API entry point | AWS API Gateway | Single URL, path-based routing to all services |
| Frontend hosting | Nginx on EKS | Multi-stage Docker build, served as a pod alongside backend services |
| SSL certificates | AWS ACM | Auto-renewing; one cert on API Gateway |
| Container registry | ECR | One repository per service + frontend |
| Secrets | AWS Secrets Manager + External Secrets Operator | ESO syncs secrets into Kubernetes |
| Admin access | SSM Session Manager | No Bastion host, no open SSH ports |

### Network Security

- **VPC** with public/private subnet separation
  - Public: ALB, API Gateway endpoints
  - Private: EKS nodes, RDS, SQS VPC endpoints
- **Kubernetes NetworkPolicies** enforce pod-to-pod microsegmentation — a compromised service cannot reach another service's database
- **IRSA (IAM Roles for Service Accounts)** — each service has its own IAM role with least-privilege permissions (its own DB + secrets only)

## Deployment & CI/CD

### Environments

| Environment | Purpose | RDS size | EKS nodes |
|---|---|---|---|
| dev | Active development | db.t3.micro | 1 |
| qa | Nightly automated testing | db.t3.micro | 1 |
| uat | Stakeholder validation | db.t3.small | 2 |
| prod | Live | db.t3.medium | 3+ |

### Promotion Flow

```
Dev push
  → GitHub Actions: lint → unit tests → integration tests → build image → push to ECR → deploy to dev (rolling)

2AM nightly (cron)
  → Promote dev image → QA → run E2E tests (pytest + Playwright)

Conventional commit (RC1, RC2, ...) or PR merge to main
  → Promote QA-approved image → UAT → smoke tests

git tag v1.0.1 + git push origin v1.0.1
  → Run Alembic migrations on prod RDS
  → ArgoCD Blue/Green: deploy Green (new version)
  → Health checks pass → swap 100% traffic Blue → Green
  → Blue pods drain (15 min) → terminate
  → GitHub Release created
```

> Manual "click-to-deploy" in the AWS Console is prohibited. All promotions are git-event driven.

### CI/CD Tooling

- **GitHub Actions** — CI pipeline (test, build, push ECR, image tagging)
- **ArgoCD** — GitOps CD; watches the repo and syncs Helm chart changes to EKS
- **Helm** — Kubernetes manifest templating; one chart per service, one values file per environment

### Deployment Strategy

- **Dev / QA / UAT:** Kubernetes rolling updates (`maxUnavailable: 0`, readiness probe gated)
- **Production:** Blue/Green deployment via ArgoCD Rollouts
  - Green is deployed and health-checked before traffic switches
  - Blue kept alive for 15 minutes as instant rollback target
  - All Alembic migrations run before any pod is replaced

### Image Tagging Strategy (ECR)

Each image carries multiple tags to track its promotion state:

```
auth-service:a1b2c3d       ← immutable commit SHA
auth-service:dev-latest    ← latest dev build
auth-service:qa-latest     ← latest QA-promoted build
auth-service:RC1           ← release candidate
auth-service:v1.0.1        ← production release
```

### Terraform State

- **Backend:** S3 bucket + DynamoDB table for state locking
- **Structure:** Separate state file per environment (dev/qa/uat/prod)
- **Bootstrap:** One-time manual run of `terraform/bootstrap/` creates the S3 bucket and DynamoDB table before all other Terraform runs

### Testing Strategy

| Layer | Tool | When it runs | Environment |
|---|---|---|---|
| Unit | pytest | Every dev push | GitHub Actions runner |
| Integration | pytest + Docker PostgreSQL | Every dev push | GitHub Actions runner |
| E2E (backend) | pytest | After nightly QA promotion | Live QA environment |
| E2E (frontend) | Playwright | After nightly QA promotion | Live QA environment |

> E2E tests run against the live QA environment — real services, real RDS, real SQS. Test data is managed via pytest fixtures (setup/teardown per test).

### Zero Downtime Guarantees

| Concern | Solution |
|---|---|
| Rolling AMI updates | ALB connection draining + ASG rolling replacement |
| Pod replacement | `maxUnavailable: 0` + readiness probes |
| Production deploys | Blue/Green — no traffic switch until Green is healthy |
| Schema changes | Alembic expand/contract — additive migrations before deploy, destructive after |
| SQS during deploy | Messages requeue after visibility timeout if consumer restarts |

## Observability & Logging

### How Monitoring Works

```
EKS Nodes
  └── node-exporter DaemonSet        ← exposes CPU/memory/disk metrics on :9100
  └── kube-state-metrics Deployment  ← exposes Kubernetes object metrics

Prometheus (StatefulSet)
  └── scrapes node-exporter + kube-state-metrics every 15s via ServiceMonitors
  └── evaluates PrometheusRules → fires alerts to Alertmanager
  └── retains 15 days of metrics on a 20 Gi EBS volume

Alertmanager
  └── routes alerts → AWS SNS topic → email subscribers

Grafana (Deployment, https://grafana.cs686.live)
  └── Prometheus datasource → metrics dashboards
  └── Loki datasource        → log explorer

Loki (StatefulSet)
  └── receives logs from Promtail DaemonSet (one pod per node)
  └── Promtail ships stdout of all pods via Kubernetes log discovery
```

All components are deployed via the `kube-prometheus-stack` Helm chart plus a standalone Loki Helm chart, both in the `monitoring` namespace.

### Metrics & Dashboards

- **Prometheus** — scrapes metrics from all EKS nodes and services via `ServiceMonitor` CRDs
- **Grafana** — pre-provisioned dashboard: **Infrastructure → node-exporter-full** (gnetId 1860) shows per-node CPU, memory, disk, and network
- **Grafana access:** https://grafana.cs686.live — login via GitHub OAuth (username/password login disabled)

### Alert Thresholds

Defined in `DevOps/k8s/monitoring/node-alerts.yaml` as a `PrometheusRule`. All alerts must persist for 5 minutes before firing.

| Metric | Warning | Critical |
|---|---|---|
| CPU usage | > 80% | > 90% |
| Memory usage | > 85% | > 95% |
| Disk usage (/) | > 80% | > 90% |

- **Warning** alerts go to the `default` Alertmanager receiver (SNS → email)
- **Critical** alerts go to the `critical` Alertmanager receiver (same SNS topic, separate routing)

To subscribe an email address to alerts:
```bash
aws sns subscribe \
  --topic-arn arn:aws:sns:us-west-2:793012580999:fitness-tracker-dev-alerts \
  --protocol email \
  --notification-endpoint <your-email> \
  --region us-west-2
```
AWS sends a confirmation email — click the link to activate.

To apply the alert rules to the cluster:
```bash
kubectl apply -f DevOps/k8s/monitoring/node-alerts.yaml
```

### Centralized Logging

- **Loki** — log aggregation (label-indexed, not full-text), stores logs on a 10 Gi EBS volume
- **Promtail** — DaemonSet on every EKS node, ships stdout of all pods to Loki
- **Grafana Explore** — query logs alongside metrics in the same UI

All 3 services log to stdout. Kubernetes captures stdout. Promtail collects by pod label.

Cross-service log query (LogQL):
```
{app=~"auth-service|workouts-service|analytics-service"} |= "ERROR"
```

Per-service query:
```
{app="workouts-service"} | json | level="ERROR"
```

## Post-Rebuild Checklist

After every `terraform destroy` + `terraform apply` cycle, complete these steps in order:

### 1. Bootstrap Terraform state bucket
Creates the S3 bucket and DynamoDB lock table used by all environments.
**All commands use absolute paths — run from anywhere:**
```bash
cd ~/cs686/final/cs686-Final-Source/DevOps/terraform/bootstrap
terraform init
terraform apply -auto-approve
```

### 2. First terraform apply (creates ECR repos, will fail on Lambdas — expected)
```bash
cd ~/cs686/final/cs686-Final-Source/DevOps/terraform/environments/dev
terraform init
terraform apply -var-file=/home/nfnewbury-dev/dev.tfvars -var="alb_dns_name=" -auto-approve
```
This will fail with `Source image does not exist` for the Lambda functions. That's expected — ECR repos are now created.

### 3. Push test-runner image to ECR
```bash
cd ~/cs686/final/cs686-Final-Source
aws ecr get-login-password --region us-west-2 | docker login --username AWS --password-stdin 793012580999.dkr.ecr.us-west-2.amazonaws.com
docker build --platform linux/amd64 --provenance=false -f backend/tests/Dockerfile -t 793012580999.dkr.ecr.us-west-2.amazonaws.com/fitness-tracker/test-runner:latest .
docker push 793012580999.dkr.ecr.us-west-2.amazonaws.com/fitness-tracker/test-runner:latest
```

### 4. Update ACM certificate ARN
The bootstrap creates a new certificate each rebuild. Get the new ARN and update it in two places:

```bash
cd ~/cs686/final/cs686-Final-Source/DevOps/terraform/bootstrap
terraform output acm_certificate_arn
```

Update these files with the new ARN:
- `DevOps/terraform/environments/dev/dev.tfvars` → `acm_certificate_arn`
- `DevOps/k8s/monitoring/kube-prometheus-stack-values.yaml` → `aws-load-balancer-ssl-cert` annotation

### 5. Second terraform apply (creates Lambdas)
```bash
cd ~/cs686/final/cs686-Final-Source/DevOps/terraform/environments/dev
terraform apply -var-file=dev.tfvars -var="alb_dns_name=" -auto-approve
```

### 6. Run bootstrap-cluster.sh
```bash
~/cs686/final/cs686-Final-Source/DevOps/scripts/bootstrap-cluster.sh dev
```

### 7. Set gp2 as the default storage class
EKS creates the `gp2` storage class but does not mark it as default. Prometheus and Loki PVCs will stay `Pending` without this:
```bash
kubectl patch storageclass gp2 -p '{"metadata": {"annotations":{"storageclass.kubernetes.io/is-default-class":"true"}}}'
```

### 8. Apply Prometheus alert rules
```bash
kubectl apply -f DevOps/k8s/monitoring/node-alerts.yaml
```

### 9. Re-subscribe email to SNS alerts
The SNS topic is destroyed and recreated on each rebuild, so subscriptions are lost. Re-subscribe after each rebuild:
```bash
aws sns subscribe \
  --topic-arn arn:aws:sns:us-west-2:793012580999:fitness-tracker-dev-alerts \
  --protocol email \
  --notification-endpoint newbury.ndewedo@gmail.com \
  --region us-west-2
```
Check inbox and click the confirmation link.

### 10. Trigger CI to push service images
gh secret set AWS_ACCESS_KEY_ID --body "$AWS_ACCESS_KEY_ID"
gh secret set AWS_SECRET_ACCESS_KEY --body "$AWS_SECRET_ACCESS_KEY"
gh secret set AWS_SESSION_TOKEN --body "$AWS_SESSION_TOKEN"

Push an empty commit or use the GitHub Actions UI to run the CI workflow. ArgoCD will pick up new images automatically.

### 11. Verify Grafana DNS
```bash
# Check current ELB
kubectl get svc -n monitoring kube-prometheus-stack-grafana -o jsonpath='{.status.loadBalancer.ingress[0].hostname}'

# Check Route53 record
aws route53 list-resource-record-sets --hosted-zone-id Z04986691K8ZU67Z9P8WB \
  --query "ResourceRecordSets[?Name=='grafana.cs686.live.']"
```

### 12. Scale node group to 3 nodes if monitoring pods stay Pending
t3.medium nodes hold a maximum of 17 pods each. With 2 nodes, monitoring pods (Prometheus, Loki) may not schedule. Scale up if needed:
```bash
aws eks update-nodegroup-config \
  --cluster-name fitness-tracker-dev \
  --nodegroup-name fitness-tracker-dev-nodes \
  --scaling-config desiredSize=3,minSize=1,maxSize=4 \
  --region us-west-2
```

If a pod stays Pending with `volume node affinity conflict`, the PVC was provisioned in a different AZ than the available node. Delete the PVC and pod so it rebinds in the right AZ:
```bash
kubectl delete pvc <pvc-name> -n monitoring
kubectl delete pod <pod-name> -n monitoring
```

### 13. Clear stale ReplicaSets if pods are stuck Pending
After CI runs and ArgoCD re-syncs, old RSes from the first (failed) sync may fill node capacity:
```bash
# Find stale RSes (old SHA, desired > 0)
kubectl get rs -n dev -o custom-columns='NAME:.metadata.name,IMAGE:.spec.template.spec.containers[0].image,DESIRED:.spec.replicas'

# Delete any RS using a SHA that no longer exists in ECR
kubectl delete rs -n dev <old-rs-name>
```

### 13. Recover from Terraform state lock (expired Voclabs credentials)
If credentials expire mid-apply, Terraform leaves a stale lock:
```bash
# Get the lock ID from the error message, then:
terraform force-unlock -force <LOCK_ID>

# If errored.tfstate exists, push it first:
terraform state push errored.tfstate
```




Before running destroy, a few things to be aware of:

EBS volumes — the Prometheus and Loki PVCs will be deleted. The EBS volumes have ReclaimPolicy: Delete so they'll be gone too.
RDS — all databases and data will be destroyed.
ECR images — these persist independently of Terraform, so they'll survive.
Terraform state lock — if credentials expire mid-destroy (as happened before), you'll need to force-unlock again.
